### 🔴 S1-A1: Space/F5 키 핸들러가 TextField 입력 가로채기

- **우선순위**: P0 (사용자 입력 차단)
- **대상 파일**: `lib/screens/song_list_screen.dart`
- **대상 위치**: `_handleKeyEvent` 메서드 (`:78-94`)

#### 문제
곡 등록/수정 다이얼로그의 제목 `TextField`에 한글/공백을 입력하려 할 때, Space 키를 누르면 전역 키 핸들러가 이를 가로채 **재생이 토글**됨. 텍스트 입력이 불가능.

#### 작업 지시
1. `_handleKeyEvent` 시작 부분에 텍스트 입력 포커스 검사를 추가한다.
2. `FocusManager.instance.primaryFocus`가 `EditableText` 계열 위젯을 가리키면 핸들러가 `false`를 반환하도록 한다.

#### 구현 예시
```dart
bool _handleKeyEvent(KeyEvent event) {
  if (event is! KeyDownEvent) return false;
  if (!mounted) return false;

  // 텍스트 입력 중에는 단축키 비활성화
  final primaryFocus = FocusManager.instance.primaryFocus;
  if (primaryFocus?.context?.widget is EditableText) {
    return false;
  }

  final key = event.logicalKey;
  if (key == LogicalKeyboardKey.space) {
    _togglePlayPause();
    return true;
  }
  if (key == LogicalKeyboardKey.f5) {
    final song = _selectedSong;
    if (song != null) {
      _openPrompter(song);
    }
    return true;
  }
  return false;
}
```

#### 완료 기준 (DoD)
- [ ] 곡 등록 다이얼로그 제목 입력창에서 Space 입력 시 재생 토글이 발생하지 않는다
- [ ] 곡 수정 다이얼로그 제목 입력창에서 동일하게 동작한다
- [ ] 다이얼로그 종료 후 메인 화면에서 Space/F5 단축키가 정상 동작한다

#### 테스트 방법
1. 곡 등록 버튼 클릭 → 제목 입력창에 "가 나 다" 입력 → 정상 입력 확인
2. 다이얼로그 닫기 → 메인에서 Space → 재생 토글 확인
3. `flutter analyze` 통과

### 🔴 S1-A2: 큐 드래그 재정렬 미동작

- **우선순위**: P0 (README/명세 불일치)
- **대상 파일**: `lib/screens/song_list_screen.dart`
- **대상 위치**: `_buildQueuePanel` 메서드 (`:1519-1652`)

#### 문제
README(`:9`)와 `docs/기능명세서.md`에 "예약 큐 ... 드래그 재정렬"이 명시되어 있으나, 현재 `GridView.builder`(`:1571`)에 reorder 콜백이 없고 `NeverScrollableScrollPhysics`가 적용되어 **드래그로 순서 변경이 불가능**.

#### 작업 지시
1. 의존성 추가 검토: `pubspec.yaml`에 `reorderables` 패키지(또는 유사) 추가, 또는 Flutter 내장 `ReorderableWrap`/`ReorderableListView` 활용 검토.
2. `_buildQueuePanel`의 격자 영역을 드래그 재정렬 가능한 위젯으로 교체한다.
3. 재정렬 콜백에서 `_queue` 순서를 갱신하고 `saveQueue` 호출 후 `setState`.

#### 구현 예시 (내장 ReorderableWrap 미지원 시 패키지 또는 세로 ReorderableListView 전환)
```dart
// 방안 A: 세로 리스트로 전환 (가장 단순)
ReorderableListView(
  buildDefaultDragHandles: false,
  onReorder: (oldIndex, newIndex) async {
    if (newIndex > oldIndex) newIndex -= 1;
    final next = List<QueueItem>.from(_queue);
    final item = next.removeAt(oldIndex);
    next.insert(newIndex, item);
    await _repo.saveQueue(next);
    if (mounted) setState(() => _queue = next);
  },
  children: [
    for (var i = 0; i < _queue.length; i++)
      _buildQueueTile(i, key: ValueKey('queue_$i')),
  ],
);
```

- [ ] 재정렬 후 앱 재시작 시에도 변경된 순서가 유지된다
- [ ] "비우기" 버튼과 삭제 버튼은 기존대로 동작한다

#### 테스트 방법
1. 곡 3개 예약 → 중간 항목을 첫 번째로 드래그 → 순서 변경 확인
2. 앱 종료 후 재실행 → 순서 유지 확인
3. 현재 재생 중인 곡이 완료된 후 다음 큐 순서대로 재생되는지 확인

- 격자(3열) 유지가 필요하면 `reorderables` 패키지 도입 필수 → 의존성 증가

### 🟠 S1-A4: 가사 전용 곡 큐 자동재생 무한 대기

- **우선순위**: P1 (핵심 기능 결함)
- **대상 파일**: `lib/screens/song_list_screen.dart`
- **대상 위치**: `_onSongCompleted` (`:153-161`), `_playNextFromQueue` (`:163-186`)

#### 문제
큐 자동재생은 `PlayerState.completed` 이벤트에 의존. 가사 전용 곡(반주 없음)은 재생이 시작되지 않으므로 `completed`가 발생하지 않아 **다음 큐로 넘어가지 못하고 무한 대기**.

#### 작업 지시
1. `_loadSong`에서 반주가 없는 곡(`availableTrackSlots.isEmpty`)을 로드한 경우, 큐에 다음 항목이 있으면 **타이머 기반 자동 스킵** 처리를 추가.
2. 반주 없는 곡은 가사 표시 후 N초(기본 3초, 설정 가능) 후 다음 큐로 진행.
3. 또는 "가사만 표시 후 수동 다음" 모드 제공. (사용자 경험 검토 필요)

#### 구현 예시
```dart
Timer? _noAudioSkipTimer;

Future<void> _loadSong(Song song, {int? preferredSlot, bool autoPlay = false}) async {
  // ... 기존 로직 ...

  await _repo.saveLastSongId(song.id);
  await _prepareAudioForSelection();

  // 반주가 없는 곡이고 큐가 있으면 자동 스킵 예약
  _noAudioSkipTimer?.cancel();
  if (song.backingTracks.isEmpty && _queue.isNotEmpty) {
    _noAudioSkipTimer = Timer(const Duration(seconds: 5), () {
      if (mounted) _onSongCompleted();
    });
  }

  if (autoPlay && _audioReady) {
    await _player.seek(Duration.zero);
    await _player.resume();
  }
}

// dispose()에서 _noAudioSkipTimer?.cancel() 추가
```

#### 완료 기준 (DoD)
- [ ] 반주 없는 곡이 큐 맨 앞일 때, 일정 시간 후 다음 큐로 자동 진행된다
- [ ] 수동으로 다음 곡을 넘길 수 있는 방법(버튼 또는 단축키)이 있다
- [ ] 타이머가 dispose 시 취소된다 (메모리 누수 없음)

#### 테스트 방법
1. 반주 없는 곡 A와 반주 있는 곡 B를 순서대로 큐에 추가
2. A 로드 → 5초 후 B로 자동 전환 확인
3. 전환 중 화면 회전/백그라운드 전환 시 크래시 없음 확인

#### 리스크
- 자동 스킵 시간이 너무 짧으면 가사를 읽을 수 없음 → 사용자 설정으로 조절 가능하도록 (Sprint 4 D항과 연계)

### 🟠 S1-A5: 가사 전용 곡 자동 스크롤 미동작

- **우선순위**: P1 (저시력 사용자 핵심 기능)
- **대상 파일**: `lib/screens/song_list_screen.dart`
- **대상 위치**: `_syncAutoScroll` (`:327-344`)

#### 문제
`_syncAutoScroll`이 `_playing` 게이트로 묶여 있음. 반주가 없으면 재생이 시작되지 않으므로 **가사 자동 스크롤 불가**. 저시력 무대 환경에서는 가사만 표시하며 자동 스크롤이 필요한 경우가 많음.

#### 작업 지시
1. `_syncAutoScroll`의 시작 조건을 `_playing`에서 `_playing || _settings.speedLevel > 0`로 완화.
2. 단, 반주 재생 중에는 재생 진행과 스크롤이 연동되도록 우선순위 유지.
3. 반주 없는 곡에서도 속도 슬라이더가 0보다 크면 자동 스크롤 동작.

#### 구현 예시
```dart
void _syncAutoScroll() {
  _autoScrollTimer?.cancel();
  // 반주 재생 중이거나, 속도 설정이 있으면 자동 스크롤
  final shouldScroll = _playing ||
      (_selectedSong != null &&
       _selectedSong!.backingTracks.isEmpty &&
       _settings.speedLevel > 0);
  if (!shouldScroll ||
      _settings.speedLevel <= 0 ||
      !_lyricsScrollController.hasClients) {
    return;
  }
  // ... 기존 타이머 로직 ...
}
```

#### 완료 기준 (DoD)
- [ ] 반주 없는 곡에서 속도 슬라이더 > 0일 때 가사가 자동 스크롤된다
- [ ] 반주 있는 곡은 기존대로 재생과 연동된다
- [ ] 속도 슬라이더를 0으로 하면 스크롤이 멈춘다

#### 테스트 방법
1. 반주 없는 곡 로드 → 속도 슬라이더를 3으로 설정 → 가사 자동 스크롤 확인
2. 스크롤 중 속도 0 → 스크롤 정지 확인
3. 반주 있는 곡 재생 → 기존 동작 유지 확인

### 🟠 S1-A6: `catch (_)` 예외 삼킴 정정

- **우선순위**: P1 (데이터 유실 위험)
- **대상 파일**: `lib/repository/song_repository.dart`, `lib/screens/song_list_screen.dart`

#### 문제
로드 실패 시 `catch (_) { return []; }` 패턴이 repository 전역에 산재. 파일 손상·권한 오류·인코딩 문제가 사용자에게 **아무 알림 없이 데이터 유실**로 나타남.

#### 대상 위치 (grep 결과)
- `song_repository.dart:89` (`loadSongs`)
- `song_repository.dart:105` (`loadSettings`)
- `song_repository.dart:121` (`loadQueue`)
- `song_list_screen.dart:470, 478, 511, 549, 769, 776` (다양한 파일 IO)

#### 작업 지시
1. `song_repository.dart`의 각 catch 블록에서 `debugPrint` 또는 전용 로거로 에러 로깅 추가.
2. 사용자에게 알림이 필요한 경우(`loadSongs` 등)는 예외를 상위로 전파하거나, 결과에 에러 정보를 포함시켜 UI에서 스낵바 표시.
3. **에러 타입을 명시** (`catch (e, stack)`): `_` 대신 `e` 사용.

#### 구현 예시
```dart
// song_repository.dart
import 'package:flutter/foundation.dart';

Future<List<Song>> loadSongs() async {
  final prefs = await SharedPreferences.getInstance();
  final raw = prefs.getString(_kSongsKey);
  if (raw == null || raw.isEmpty) return [];
  try {
    // ... 기존 로직 ...
  } catch (e, stack) {
    debugPrint('loadSongs 실패: $e\n$stack');
    // 사용자 알림을 위해 예외 전파 또는 결과 객체 반환 검토
    rethrow; // 또는 return [] with error flag
  }
}

// song_list_screen.dart - _addSong
try {
  // ...
} catch (e) {
  debugPrint('곡 추가 실패: $e');
  _showSnack('곡 추가 중 오류가 발생했습니다: $e');
}
```

#### 완료 기준 (DoD)
- [ ] 모든 `catch (_)` 블록이 `catch (e)` 또는 `catch (e, stack)`으로 변경되었다
- [ ] 각 블록에서 최소 `debugPrint` 로깅이 추가되었다
- [ ] 사용자 영향이 큰 로드 실패(`loadSongs`)는 스낵바 또는 전용 에러 UI로 알림
- [ ] 민감한 정보(파일 경로 등)가 로그에 과도하게 노출되지 않는다

#### 테스트 방법
1. SharedPreferences 데이터를 고의로 손상(잘못된 JSON) → 앱 실행 → 스낵바 표시 확인
2. 권한 없는 경로 지정 → 에러 로그 확인(debugPrint)
3. 정상 케이스는 기존 동작 유지

#### 리스크
- 에러 메시지에 내부 경로 노출 → 보안/프라이버시 검토 필요
- `rethrow` 남용 시 앱 크래시 가능 → UI 계층에서 반드시 try-catch
