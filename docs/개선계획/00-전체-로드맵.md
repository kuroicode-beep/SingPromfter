# SingPromfter 전체 개선 로드맵

> 작성일: 2026-07-04
> 검수 기준 커밋: `ff902d9` (v0.5 beta)
> 분석 결과: `flutter analyze` ✓ No issues / 테스트: 모델·로직·위젯 20개 통과

---

## 1. 배경 및 목표

저시력 사용자를 위한 노래 프롬프터 앱 **SingPromfter**의 전체 코드 검수 결과를 반영하여,
5개 스프린트로 나누어 점진적으로 개선한다.

**핵심 목표**
1. 사용자에게 즉시 영향을 주는 버그 제거
2. 핵심 타겟인 저시력 사용자 접근성 확보
3. 유지보수 가능한 구조로 리팩토링
4. 데이터 안전성과 확장 기능 확보
5. 자동화된 품질 보증 체계 구축

---

## 2. 스프린트 개요

| 스프린트 | 주제 | 기간(권장) | 산출 | 문서 |
|---|---|---|---|---|
| **Sprint 1** | 긴급 버그 수정 | 1~2일 | 버그 6건 | `01-스프린트1-긴급버그수정.md` |
| **Sprint 2** | 접근성 강화 | 3~5일 | 접근성 5건 | `02-스프린트2-접근성강화.md` |
| **Sprint 3** | 아키텍처 리팩토링 | 5~7일 | 구조 5건 | `03-스프린트3-아키텍처리팩토링.md` |
| **Sprint 4** | 기능 확장 | 5~10일 | 기능 7건 | `04-스프린트4-기능확장.md` |
| **Sprint 5** | 품질 보증 | 3~5일 | 품질 6건 | `05-스프린트5-품질보증.md` |

> 기간은 1인 개발 기준이며, 각 스프린트는 독립적으로 진행 가능하다.
> 단, **Sprint 3(리팩토링)는 Sprint 4/5의 전제조건**이므로 순서를 지킬 것.

---

## 3. 스프린트 의존 관계

```
Sprint 1 (버그수정) ──────────────┐
                                   ├──→ Sprint 3 (리팩토링) ──→ Sprint 4 (기능확장)
Sprint 2 (접근성) ─────────────────┘                         └──→ Sprint 5 (품질보증)
```

- **Sprint 1, 2**는 서로 독립적이며 동시/선후관계 무관하게 진행 가능
- **Sprint 3**은 1·2의 결과를 통합한 구조로 리팩토링 (먼저 하면 재작업 발생)
- **Sprint 4, 5**는 리팩토링된 구조 위에서 진행

## 4. 개선 항목 분류 (검수 보고서 매핑)

### 🔴 A. 버그 / 결함 → **Sprint 1**
- A1. 글로벌 키보드 핸들러가 TextField 입력을 가로챔 (Space/F5)
- A2. 큐 드래그 재정렬 미동작 (명세 불일치)
- A3. `LyricsScreen` dead code
- A4. 큐 자동재생 - 가사 전용 곡 무한 대기
- A5. 가사 전용 곡 자동 스크롤 미동작
- A6. `catch (_)` 예외 삼킴 (데이터 유실 위험)

### 🟠 B. 아키텍처 / 구조 → **Sprint 3**
- B1. `SongListScreen` 1,970줄 God Widget 분할
- B2. 매직 넘버/로직 3중 복제 → 공통 모듈화
- B3. SharedPreferences 전체 직렬화 → 저장소 분리
- B4. Web 지원 코드 정리 (방향 결정)
- B5. 레거시 래퍼(`getMrPath`/`copyMr`) 제거

### 🟡 C. 접근성 → **Sprint 2**
- C1. 슬라이더 +/- 버튼 보조 조작
- C2. 터치 타겟 48dp 확보
- C3. Semantics / 스크린 리더 라벨
- C4. 색 대비 검증
- C5. 폰트 크기 단계 정교화

### 🟢 D. 기능 확장 → **Sprint 4**
- D1. 곡 검색 / 필터
- D2. 백업 / 내보내기 / 가져오기
- D3. 삭제 복구 (휴지통)
- D4. 반주 시작/끝 지점 (트리밍)
- D5. 반주 라벨 사용자 정의
- D6. 재생 속도 조절
- D7. 일괄 등록

### 🔵 E. 품질 / 도구 → **Sprint 5**
- E1. 단위 테스트 작성
- E2. CI 구축 (GitHub Actions)
- E3. 의존성 업데이트
- E4. analysis_options 강화
- E5. 주석 코드 정리
- E6. 매직 넘버 상수화

## 6. 스프린트 진행 규칙

각 스프린트 문서는 다음 형식을 따른다:

1. **목표**: 이 스프린트에서 달성할 것
2. **작업 항목(Task)**: 개별 수정 단위
   - ID (예: `S1-A1`)
   - 우선순위 (P0~P3)
   - 대상 파일/위치
   - 상세 작업 지시
   - 완료 기준(Definition of Done)
   - 테스트 방법
3. **리스크 및 주의사항**
4. **수용 기준(Acceptance Criteria)**
